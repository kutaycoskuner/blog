Citation Standards

Published: Tue, 18 Apr 2023 Revision: Sat, 25 Jan 2025, v 1.4 Language: This content is in English.

This is my self citation notation. I required to create this documents since I could not find both reliable syntactically coherent and citation guide.

  • Citation might fully grammatical syntactic but might take long time to write and looks ugly.
  • Citation might be elegant but consist of a lot of exceptions and problematic cases.
  • Some aspect considered, consistency, readability, extensibility and aesthetics.

Aim of this study to mix a set elegant and systatic citation setup that includes vast range of sources without question marks.

  • referenc entry
  • footnote
  • in-text reference

Types of Citation

Three types of citation is held in this work these are

  • In-text citation [WIP] : Most compact form of citation. Not favored due to preventation of readability. However it is convenient on some cases.
  • Footnote/ Endnote citation [WIP] : Often contains explanation or specific information (ex. page in book)
  • Bibliography citation : Contains full information about to source

Semantics

  1. Citation gives minimum three of five metadata information about the source:
    • Who: Who has created the work?
    • What: What is the name of work?
    • Where: If it is published where?
    • When: When it is published or settled as frozen entity in history?
    • How: How do we reach it? Access link / doi etc.
    • The question why could be answered within the text or in footnotes but it is not a part of citation.
    • ex. Who. What. Where. When. How
  2. One or information might be missing or not included. However, the order of the information is always the same and a citation should contain minimum 3 of the metadata information.
    • ex. Author. Book. Publisher. Date.
  3. Each metadata information is separated and ended by . (dot) from each other except for how.
    • ex. Author. Book. Publisher. Date.
  4. If the metadata is not conventional type: dissertation, book, article, newspaper. It should be indicated with square braces.
    • ex. [Video], [Podcast]
  5. The source of the work is written italic. It might be a chapter of book, article in journal or a book itself.
    • ex. Author. A Book . …
    • ex. Author. “Article”. In Journal. …
  6. Part of something relationships indicated through quotation marks.
    • ex. “Concept”. In Dictionary.
    • ex. “Article” In Journal.
  7. If the source name is in diffent language than article. The translation might be given in square braces after the name.
    • ex. Artikel [Article].
  8. Page numbers are categorical exception handled as how. comes after date with : separation.

Example Book Citation:

Arendt, Hannah. The Human Condition. Chicago: University of Chicago Press. 1958.

Metadata Handling

  • Who Who produced the work? Author, editor, producer etc.

    • Occasions
      • Single author
        • Desciption In bibliography reference last name is written first separated by comma (,)
        • Template Last Name, First Name.
        • Example Newton, Isaac.
        • Exception If author has more than one names, only first name is written full rest is shortened by first letter.
          • ex. Popper, Karl R.
      • Two Author
        • Desciption In bibliography reference last name is written first separated by comma (,) First name or names written shortened by dot (.) at the end.
        • Template Last Name 1, First Name First Letter 1. and Last Name 2, First Name First Letter 2.
        • Example Acemoğlu, D. and Robinson, J. A.
      • Three or more author
        • Desciption Only first author name and constant abbreviation et al. (Latin abbreviation means “and others”)
        • Template Last Name of First Author et al.
        • Example Berg et al.
      • Pseudonym
        • Desc Sometimes authors uses nicknames for anonymity. In such cases pseudonym is written as it is with (pseudonym) constant. But if the pseudonym is well known the constant might be avoided.
        • Template Pseudonym Name, (pseudonym).
        • Example Bulkhardt, Richard (pseudonym).
        • Example Le Corbusier.

  • What What is the name or title of the work?

    • Title of work
    • Part of something

  • Where the work is produced?

    • Template City: Publisher

  • When When the work is produced or published?

    • Occasions
      • Year of publish
      • Multiple publications in the same year

  • How How to find or accessed to the document

    • Links: Accessed (DD MMM YYYY)
    • Dois:

Code Standards [Draft]

Published: Tue, 18 Apr 2023 Revision: Wed, 12 Mar 2025, v 0.6 Language: This content is in English.

This document is a self-defined standard developed from various sources and experiences to ensure cohesive and consistent code quality. It considers key parameters such as readability, consistency, aesthetics, writing/editing efficiency, and platform independence.

Terminology

  • Naming Conventions

    • flatcase: myvariablename
    • snake_case: my_variable_name
    • PascalCase: MyVariableName
    • camelCase: myVariableName
    • kebab-case: my-variable-name
    • mixed_caseNaming: My_variableName

  • Letter Cases

    • lower case: all lowercase
    • upper case / capital case: ALL UPPER CASE
    • proper case / title case: All Proper Case
    • sentence case: First word capitalized.

  • Programming Languages and Abbreviations

    • C c
    • C++ cpp
    • JavaScript js
    • Python py
    • Cascading Style Sheets css

  • Brackets

    • () Parantheses
    • {} Curly braces
    • [] Square braces
    • <> Angle brackets

  • Indentation Styles

    • Allman (BSD Style)
      function do_things()
  {
  }
    • K&R Style (Kernighan & Ritchie)
      function do_things() {
  }

  • Special Character Names

    • - dash
    • ; semicolon
    • ` backtick
    • & ampersand
    • : colon
    •   whitespace
    • _ underscore

Broad Guidelines for Naming Things

  1. Use U.S. English spelling and grammar in all code and comments.

    • Good: color, initialize
    • Bad: colour, initialise

  2. Consider the reader’s perspective, not the writer’s.

    • Good: calculateDistance() - clear and meaningful
    • Bad: calcDst() - too abbreviated

  3. Avoid ambiguous names, analogies, or context-specific terms.

    • Good: totalUsers
    • Bad: stuff or data123

  4. Use abbreviations only when the meaning is clear and consistent as it helps with aligned code editing.

    • Good: vrtx for vertex
    • Bad: col for color (collision with “column”)

  5. Prepositions are not included in capitalization or words separation.

    • Good: distancetoNextNode
    • Bad: DistanceToNextNode

Broad Guidelines for Formatting

  1. Be consistent on maximum character length on rows including comments.
    Reason: this will provide a standard view for various platforms and monitors.

  2. Imports are on separate lines.
    Reason: allows code editing disabling / enabling.

    • Good: 
      import os
import sys
    • Bad: 
      import os, sys

  3. Use 4 spaces or 8 spaces for indentation.
    Reason: Passively preventing you from over-indentation; thus increasing code readability.

    • Good: 
      def my_function():
    print("Hello, world!")
    • Bad: 
      def my_function():
  print("Hello, world!")

  4. Use white space for readability.
    Reason: Increasing readability.

    • Good: 
      int result = (value1 + value2) / (total - count);
    • Bad: 
      int result=value1+value2/(total-count);

  5. Use Allman or K&D Style Indentation
    Reason: This is mainly personal preference. I find Allman much more readable but increasing code length as a trade off.

  6. Favor and use alignment for better readability.
    Reason: Code is easier to read and edit.

- Commenting and Commit messages

Commit messages

  • should be concise, descriptive, and written in lowercase separated by commas
    Examples:
    fix memory leak, optimize loop
    add validation for user input; remove unused imports

Comments

  • Should follow lowercase or sentence case.
  • Some suggested usage of comments license, metadata, citation, link, descriptive, separator, marker or visual
  1. Descriptive Comments:
  • Used to explain the purpose of code blocks, algorithms, or logic. 
    // Initialize the matrix with default values
  1. Separator Comments:
  • Used to visually group sections of code. 
    // -----------------------------------------------------------------------------------
//         Data Preprocessing Section
// -----------------------------------------------------------------------------------
  1. Marker Comments:
  • Highlights critical parts, such as TODOs or FIXMEs. Usually aimed to be searched. 
    // TODO: Add validation for empty inputs
// todo: Add validation for empty inputs
  1. Visual Description Comments:
  • Uses ASCII art or other visualization to represent concepts or improve code readability. 
    #   Cosine function visualization
#
#                *
#             *     *
#           *         *
#         *             *
#       *                 *
# ---*---*---*---*---*---*---*--> x
#     0   π/2   π    3π/2  2π

- Files and Directories

Directories / Folders

  1. Use snake_case for folder names.
    Examples:
user_profiles/
data_processing/
api_endpoints/

File names

  1. Use snake_case for simple code file names (slugs) unless they contain.
    Examples:
user_profile_handler.py
data_processing_service.cpp
api_handler.js

  1. Sometimes name of file could contain more various information about the data instead of just name or slug. In that case various metadata types should be separated by a dash (-)

    Rationale: whitespace ( ) may cause some errors on commands. Underscore(_) does not allow to jump word to word by control and is already reserved for replacing white space ( ) on slug. Examples:

1.3-axes-vrtx_shader.glsl               a name for shader file
4.1-bloom-test_scene.cpp                a code file with metadata
tex-linegrid_16-white-2k-color.jpg      a texture file metadata 
core-ui.cpp                             use dash if you are using prefix instead of directories
(fileprefix-slug-detail-resolution-type)
  1. File names could hold prefix or abbreviated keywords to sort related file together.
    Examples:
abst      abstract data types    (structs / classes)
data      data                   (models, scenes, vertices, materials input in general)
map       mapping                (how to associate data / mappings ex. shader with file names)
core      core                   (main file)
util      utililities            (self written library functions, parsers / calculators etc.)
test      testing                (test files)
  1. For static third party library folders: version number attached via ”-“
    Example(s):
glm-0.9.9.8

- Variables

  • Use snake_case for variables and constants.
    Reason of choice over camelCase better readability and writing efficiency. Tradeoffs: long naming, unable to provide categorization on names.

    Single character prefixes avoided to use because it has collision for certain naming.

    Examples:

      int user_age = 25;

  • Certain data types could be marked with prefixes. But not with single characters due to collision on naming such as v_tex_coords (vertex texture coordinates) g_buffer etc.
    Examples:

  global              : gl_variable                         (important for distinction)
  bools               : is_visible, has_alpha, use_shadows  (has verb that answers yes/no question)
  constants           : VARIABLE                            (CAPITALIZED_SNAKE_CASE)
  pointers            : ptr_variable                        (important for distinction)
  const char pointer  : PTR_VARIABLE                        (combination of const and pointer)       
  private variable    : _private                            (underscore as prefix to indicate private variable)

— Constants

  • Use snake_case with a leading underscore for private members.
    Examples: 
    const int MAX_LIGHT_COUNT = 100;

— Private Members

  • Use snake_case with a leading underscore for private members.
    Examples: 
    class UserProfile {
private:
  std::string _name;
};

— Enums

  • Use UPPER_SNAKE_CASE for enum values.
    Examples: 
    enum class Color {
  RED,
  GREEN,
  BLUE
};

- Functions

  • Use snake_case for naming functions.
  • Starts with verb, as functions reflect some action.
  • Try to describe what function does precisely and shortly as much as possible.
  • Avoid ambiguities, meaning dualities or names with context. Examples: 
    void calculate_total(int price, int quantity) {
  return price * quantity;
}

- Data Structures: Classes, Structs

  • Use PascalCase for data structures.
  • Try to use phrases rather than single words if you are not using namespaces to avoid possible collisions.
    Examples: 
    class UserProfile {
public:
  UserProfile(std::string name) : name(name) {}
private:
  std::string name;
};

Asset Naming Standards [Draft]

Published: Sun, 6 Oct 2024 Revision: Tue, 28 Jan 2025, v 0.3 Language: This content is in English.

A guideline for organizing assets in digital projects.

Broad Guidelines for Asset Naming

  1. To prevent capitalization based problems and ensure platform independency use all lowercase notation for naming asset files.

    Examples:

    • Good: potato
    • Bad: Potato pOtAtO POTATO

  2. The unique identifier for an asset under categories called identifier or slug.

  • If slug has multiple words, they should be separated by dash (_) instead of white space ( ).

  • If slug has collision with an other name, it should receive incremental two digit number followed by slug (slug01). Two digit is to ensure correct sorting on computers.

  • If the distinction between two assets are too small it should receive a variation letter at the end of number instead of increment.

  • A slug may consists of just name, name + two digit number or name + two digit number + variation letter.

    Examples:

    • Good: potato-bags potato-bags-01 potato-bags-01a
    • Bad: potato_bags potato-bags-2 potato-bags-001

  1. Name of the file may contain various metadata information about the asset. Each metadata type should be separated by an underscore (_) and should be sorted in an order.

    Pattern: * required : optional

    • *[identifier] - :[detail] - :by-[author] - :[source] - :[license] . :[format]
      • Detail (optional) A human identifiable identifier name for the object.
        • Detail information : Such as block, hp (high poly), lp (low poly)
        • Style : voxel, stylized, realistic
        • Resolution : 512, 1k, 2k, 3k, 4k (required for texture or texture using objects)
        • Level of Detail : lod1, lod2, lod3
        • Part : Part of combination xp xn (x positive, x negative), color (color map for textures), ao (ambient occlusion for textures)
      • Author (optional) name of the author or owner recognized by by- tag could be person, organization or institute
      • Source (optional) name of the source tag could be person, organization or institute
      • License (optional) If asset has a commercial, attribution or preventing modification license it should be encapsulated within a folder with the license files.

    Examples:

    • Good: potato-bags-01a_2k_by-berk-gedik_sketchfab.gltf linegrid-16_darkgray_2k_color.png nebula-blue_2k_xp.jpg
